浅谈编程中的抽象

前段时间和wwj718关于抽象这一块聊了许多，我们都认为抽象对于软件开发很重要，今天我便来简单聊一聊编程中的抽象。

引言

首先，基本的抽象指的是什么？

维基百科里对它的定义是：在哲学里，“抽象化”是一种将观念抽离原本客体的思想过程。我不打算在这展开讨论哲学问题，仅以此为引子。罗素说，人们对于客观存在的认识其实是一种感觉材料，这是什么意思呢？譬如当我们谈论光的时候，我们谈论的是肉眼所见的光芒时的感觉，这是大多数人对光的认识，所以人们一般不会将光看作一种波动，尽管我们可以通过物理学的描述将它述说的很具体，也就是说客观存在和感觉材料是两个范畴，他们之间有一个相互的映射关系，我们能确定的只有自己的感觉材料，至于客观实体则不可知。

那么，这与编程有什么关系呢？问得好，简单来说就是表示与实现的分离，什么是表示？表示一般代表对外的函数名或者说接口，就像上文提到的感觉材料，站高一个抽象层面的程序员往往不需要关注它的实现部分（客观实体），实现则是这个表示对应的实现方式，有不止一种可能。比如实现一个符号表，你既可以使用链表，也可以通过哈希，甚至可以通过高阶函数来实现；再比如优先队列，你既可以通过有序或无序数组，也可以通过二叉堆来实现，不同实现对外的接口都是一样的，唯一的区别在于实现不同，性能和效率会有差异。

现在我们站在行为和数据的角度来谈论编程中的抽象：

行为抽象

如果你在不同的地方有两段同样的代码做一样的事情，那么你应该将它组织成一个函数，这样做有一个显而易见的好处，如果以后你需要调整某一功能，你只需要调整一处而非全部，同时如果发现了bug也可以避免bug的复制，简而言之就是便于单点控制，避免了维护代码的痛苦。

下面我以函数式语言中的三种基本抽象结构来展示抽象的过程，分别是map，filter，fold(r)，确实这三种结构很简单，很多FP的101课程都会介绍它们，但是我还是推荐你看一下这三种结构的推导，因为它们阐述了一些重要的原则。

map

(: negative-all : (Listof Boolean) -> (Listof Boolean))
(define (negative-all lst)
  (cond
    [(null? lst) empty]
    [else (cons (not (car lst))
                (negative-all (cdr lst)))]))

(: length-all : (Listof String) -> (Listof Integer))
(define (length-all lst)
  (cond
    [(null? lst) empty]
    [else (cons (string-length (car lst))
                (length-all (cdr lst)))]))

这两个函数都是对列表的操作，negative-all将列表中所有的Bool值都取反，length-all将列表中每一个字符串都取长，观察它们的异同我们可以发现，唯一的区别就在于递归过程中对(car lst)的操作不同。我们来手动展开negative-all, length-all的求值过程：

(negative-all (list false true))
= (cons (not false) (negative-all (list true)))
= (cons true (cons (not true) (negative-all empty)))
= (cons true (cons false empty))
= (list true false)

(length-all (list "ab" "a"))
= (cons (string-length "ab") (length-all (list "a")))
= (cons 2 (cons (string-length "a") (length-all empty)))
= (cons 2 (cons 1 empty))
= (list 2 1)

由展开过程我们可以看出这两个函数都是对一个列表中的每个元素进行某种操作后，再按原列表的顺序进行聚合。而这个某种操作在negative-all中是not，在length-all中是string-length。我们是否可以将这个某种操作通用化，在需要取反时，我们指明not，需要对字符串求长时，指明string-length…以这个思路我们应该将not和string-length视为某种功能函数f，然后将它提取到函数的参数中。

(: negative-all : (Boolean -> Boolean) (Listof Boolean) -> (Listof Boolean))
(define (negative-all f lst) ;; f 传参为 not
  (cond
    [(null? lst) empty]
    [else (cons (f (car lst))
                (negative-all f (cdr lst)))]))

(: length-all : (String -> Integer) (Listof String) -> (Listof Integer))
(define (length-all f lst) ;; f 传参为 string-length
  (cond
    [(null? lst) empty]
    [else (cons (f (car lst))
                (length-all f (cdr lst)))]))

将f提取为参数后，你会发现negative-all和length-all的body部分除了函数名以外完全一样，因此，我们可以将函数重命名为map

(: map : (All (a b) (a -> b) (Listof a) -> (Listof b)))
(define (map f lst)
  (cond
    [(null? lst) empty]
    [else (cons (f (car lst))
                (map f (cdr lst)))]))

对类型不熟悉的同学可以暂时忽略第一行类型声明，到此我们就得到了一个比较通用的列表操作map，下面是以map重新定义的negative-all和length-all

(define (negative-all lst) (map not lst))
(define (length-all lst) (map string-length lst))

filter

(: remove-not-odds : (Listof Integer) -> (Listof Integer))
(define (remove-not-odds l)
  (cond
    [(null? l) empty]
    [else (cond
            [(not (odd? (car l))) (remove-not-odds (cdr l))]
            [else (cons (car l) (remove-not-odds (cdr l)))])]))

(: remove-not-pigs : (Listof Symbol) -> (Listof Symbol))
(define (remove-not-pigs l)
  (cond
    [(null? l) empty]
    [else (cond
            [(not (eq? (quote pig) (car l))) (remove-not-pigs (cdr l))]
            [else (cons (car l) (remove-not-pigs (cdr l)))])]))

remove-not-odds移除列表中所有非odd数，remove-not-pigs移除列表中所有的非pig元素，和上一小节一样，我们先找出它们的不同点

(odd? (car l))
(eq? (quote pig) (car l))

你会发现不同点主要是递归中对列表元素的判断，以及函数名。那么我们可以将odd?和(lambda (a) (eq? a (car l)))看作一种匹配判断的功能函数，并将它抽象为函数的一个参数f，并且将函数重命名为filter，最后我们得到

(: filter : (Symbol -> Boolean) (Listof Symbol) -> (Listof Symbol))
(define (filter f l)
  (cond
    [(null? l) empty]
    [else (cond
            [(not (f (car l))) (filter f (cdr l))]
            [else (cons (car l) (filter f (cdr l)))])]))

filter函数用于列表的过滤，下面我们用filter重新实现remove-not-odds和remove-not-pigs

(define (remove-not-odds lst) (filter odd? lst))
(define (remove-not-pigs lst) (filter (lambda (a) (eq? a (car l))) lst))

foldr

(define (sum l)
  (cond
    [(null? l) 0] ;列表为空，则和为0
    [else (+ (car l) (sum (cdr l)))]))

(define (prod l)
  (cond
    [(null? l) 1] ;列表为空，则积为1
    [else (* (car l) (prod (cdr l)))]))

sum和prod两个分别为列表求和，求积。我们看出它们的不同在于对空列表的响应，以及else这一行的操作符，一个是加号，一个是乘号，而这些都是语义相关的，所以我们将它们抽象出来，0和1对应base，操作符对应组合函数combine，那么我们会得到fold的定义

(define (fold combine base l)
  (cond
    [(null? l) base]
    [else (combine (car l) (fold combine base (cdr l)))]))

然后我们可以通过fold来重新定义sum和prod

(define (sum l) (fold + 0 l))
(define (prod l) (fold * 1 l))

这里留个思考题，用fold定义前面的map和filter两个函数。

数据抽象

数据抽象大家平时基本都在做，因此我仅举一个比较实际的例子,不多做阐述，例如在知乎中，不只是回答有评论，文章和问题都有评论，这种情况下重复定义每一种评论的类型是非常不优雅的，因为每种评论除了依附的实体不同外，它们的结构几乎都相同（这里假设同一个应用内评论功能不做区分），那么我们应该定义一种抽象类型，然后将具体的类型作为它的子类型

(define-type Comment (U QuestionComment AnswerComment NoteComment))

总结

一般而言，抽象在实际的开发中总是与重构息息相关的，但很多程序员往往写不出好的抽象结构，导致代码结构过于呆滞，不方便进一步功能扩展。这里先给出重构的定义：将重复的代码重写为语义相等的函数，便于单点控制。那么这里要注意什么原则呢，从map，filter和fold的例子中，可以看出，我们应当剥离出语义相关的部分作为参数，这样便可以通过不同的参数组合出多种多样的功能。好的抽象不是凭空得来的，靠的日复一日的思考与总结。当然，你问我有没有一些捷径，我的答案是有…如果你想快速建立对抽象的认识，可以去了解抽象解释及范畴论相关的知识。
