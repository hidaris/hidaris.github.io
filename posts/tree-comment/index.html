如何实现一个知乎式的评论系统

知乎现在有这么高的人气，一个好的评论系统是功不可没的，优秀的评论系统应该方便用户进行交流。出于方便交流的目的，评论系统应当采用能够保存上下文的数据结构来存储信息。

目前比较知名的评论系统有两种，一种是reddit使用的被动呈现方式，外观上可将其视为一颗向左侧倒下的树，它将评论的上下文以树的形式完整的呈现给用户，这么做的好处是用户可以一目了然的查看到全部的交流内容，并且由于树形展示有层次感，用户不会迷失在大量的内容里。具体可见下图：

这种做法的弊端在于一旦讨论的上下文过长，以用户浏览器的尺寸是无法全部显示内容的，并且一次UI的渲染需要加载全部数据，在性能上没有优势。

reddit对此的解决方案是将过长的讨论截断，并把剩余内容调整为主动呈现方式，由用户自己点击后加载

另一种做法是知乎式的主动呈现方式，需要用户主动点击查看会话的上下文，这种方式需要实现两个API，一个查看某条评论的下层回复，一个由当前回复回溯到顶层评论。以查看对话为例, 一个非顶层评论会允许用户点击查看对话按钮：

当用户点击按钮后给出一个弹窗，列出从该回复到顶层评论的全部上下文

如果你足够细心，会发现图中查看对话和我前文描述的有些许不同，我前面说查看对话只用于非顶层评论，顶层评论只有查看下层回复的功能，但是按知乎现在的做法你可以直接从顶层评论查看整个对话的上下文，如果将评论视为一颗树的话，知乎的这种做法可能会有问题，讨论如下。

知乎评论可能存在的问题

我们聊知乎的评论时，发现它可以从顶层评论直接取出后面的对话内容，这看起来比我们的设计更方便，因为它取对话时的方向是任意的可以从树顶开始，也可以从任意节点反向取出。那么这种做法的问题是？

首先我们看一下树形评论的结构图

从这张图中我们会发现如果从Fruit节点往后的分支有两条，也就是说它的对话其实不止一条走向，那么从Fruit向下实现查看会话功能一定会出现多种会话混合的情况。关于知乎的实现方式我有一个naive的设想，如果有熟悉知乎实现方式的朋友看到这篇文章，欢迎帮我解惑。

现在我便来聊下如何实现我们之前的设计。

实现

我们打算以树形结构来表示评论的上下文，那么问题就变成如何在sql中存储树形结构，我的做法是将表的每一个实例看作树中的一个节点，每一个节点需要存储自己的深度，父节点以及回溯的根结点的路径，以Django为例

class AbstractComment(TimestampedModel, VoteMixin):
    content = models.TextField()

    user = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        related_name='%(class)s',
        on_delete=models.CASCADE)

    parentid = models.IntegerField(blank=True)

    depth = models.IntegerField(default=1)
    path = ArrayField(models.IntegerField(), default=[])

这段python代码中我们定义了一个评论的抽象类，除了前面所说的属性，它还包含内容，作者等。有了节点的深度，路径，我们怎么实现顶层评论的查看回复接口呢，首先为了方便讨论我们定义好接口名称：

  • 查看回复 api/v1/{commentid}/reply
  • 查看对话 api/v1/{commentid}/conversation

因为查看回复是查看顶层评论的回复，那么我们只要查询全部深度为2并且父节点为该顶层评论的回复内容就行了

comment = get_object_or_404(self.queryset, pk=pk)
replies = self.queryset.filter(depth=comment.depth + 1).filter(
    reference=comment.pk)

同样，实现查看对话也不复杂，我们首先获取当前评论，并得到它到根节点的路径，然后取出路径中的对应回复，按创建时间排序

comment = get_object_or_404(self.queryset, pk=pk)
path = comment.path
conversation = self.queryset.filter(id__in=path).order_by('created_at')

这样我们就可以对于任意非顶层评论实现查看对话上下文功能了。
