消息系统的概念、设计与实现

在社交网络中，用户之间的通信方式主要分为两类，一类是直接进行交流（评论，私信），另一类则是与资源互动（点赞）并通知相关的用户（消息），今天我们来聊聊消息系统。

定义

消息系统是一种将社交网络中产生的消息推送给相关用户的系统，在这里我们可以得知，消息系统首先会记录消息，将它处理成文本形式，进而推送给用户。那么我们可以得到以下抽象

Push Message -> User

我们将消息系统视为一个推送器，将用户需要知道的消息推送给它，但是这个过程中有许多细节需要考虑，比如如何得到这个消息，消息应该处理成什么形式，如何“推送”，用户自己来索取还是系统主动推给他，这些我们将在下文一一讨论。

消息

消息的格式

这一节，我们来定义消息的格式，先举两个消息的例子

小明赞同了你的回答xxxxxx

小李在收藏夹xxx中添加了yyy

例子中包含消息的产生者（小明，小李），消息记录的行为（赞同，收藏），行为交互的对象（回答，收藏夹），行为操作的对象（yyy），我们将这些要素抽象为

产生者：Actor

行为：Verb

交互的对象： Target

操作的对象：Object

如此，我们得到了消息类型的骨架，以python为例

消息的获取

通过上一小节，我们得到了消息的骨架，填充这个骨架需要从系统的上下文里获取数据。依然以点赞为例，当小明点赞你的回答时，系统应当产生一个事件，这个事件传递生成消息所必要的信息（AVOT），然后在队列中处理这个事件，将信息传递给Message类，进而得到结构化的消息

上面这段例子，我在注释里提到了待重载，因为我们可能有很多种具体的消息类型，比如点赞，收藏，关注，评论，回复等等，这些类型在结构上很相似，因此我们可以定义一个抽象的消息类，以及一些它的子类

class UpMessage(Message):
    def verb(self):
        return "点赞了"

class FollowMessage(Message):
    def verb(self):
        return "关注了"

class StarMessage(Message):
    def verb(self):
        return "收藏了"

同样的，系统中用户可以有多个点赞对象，所以 UpMessage 还可以进一步细化

class AnswerUpMessage(UpMessage):
    def template(self):
        return f"{self.actor}{self.verb}你的回答{self.target}"

class CommentUpMessage(UpMessage):
    def template(self):
        return f"{self.actor}{self.verb}你的评论{self.target}"

消息的推送

我们需要在得到消息的模版后将它推送给相应的用户，这一步有推和拉两种方式，推比较简单，当触发消息事件时，把消息存储到每个关注者的空间下。拉则相反，用户主动去问询他相关的资源是否有新消息产生并汇总。两种方式各有利弊，推的弊端在于如果某个资源的关注用户非常庞大，那么一次推的成本便很大，同样拉的弊端则相反，如果有某个用户关注了全部的资源，那么每次他去获取消息时拉的成本便很夸张。这里我来说说推的思路。
