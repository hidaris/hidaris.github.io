django-rest-framework-jwt调试小记

最近 @dajun 给我一个任务，其中涉及到了对 drf-jwt 库以及 Django 用户验证的调试，这件事情不是很困难，但是我觉得中间探索的过程挺有意思，因此将它记录于此。

问题

由于我们的系统目前还采取审核制，也就是说用户注册之后必须得通过管理员的审核才能登录，这就得在登录失败时区分未经审核的这部分用户。我需要做的就是给这些用户单独的登录失败提示。比如你的账号尚未通过审核，请联系小助手xxx这种。听起来这是个很简单的任务，我们只需要在验证用户名密码时，判断用户是否激活，并给出单独的提示信息即可。但是我们的用户验证使用了 django-rest-framework-jwt 这个库，涉及验证的逻辑都在库代码中，按理来说库的实现者不可能不考虑这种情况，否则一定会有人写一个功能更完备的库来取代它，或许还会写一篇文章“为什么 drf-jwt 不够好”，并搞一个大新闻#5838。

探索一

首先，jwt 库是一个英文项目，提示信息想来不太可能是中文，而我在测试中发现无论是输错密码还是用户未通过审核，得到的报错消息都是用户名或密码错误，这很奇怪，但考虑到我们的系统做了前后端分离，所以我猜测前端硬编码了提示消息，查阅代码后发现果然如此：

.catch(() => {
    this.$message.error("用户名或密码错误！")
    })

探索二

那么下面要做的就是看看登录失败时接口是否对未激活用户作单独提示就可以了，结果当然是没有。这时我追踪库中是否有关于未激活用户的处理，并在 github 上搜索出这段代码

user = authenticate(**credentials)
if user:
    if not user.is_active:
       msg = _('User account is disabled.')
       raise serializers.ValidationError(msg)

    payload = jwt_payload_handler(user)

    return {'token': jwt_encode_handler(payload), 'user': user}
else:
    msg = _('Unable to log in with provided credentials.')
    raise serializers.ValidationError(msg)

我们可以看出这段代码首先调用 Django 提供的 authenticate 方法来获取 user 对象，当 user 存在时判断用户是否激活，并给出提示，而既然这里没有执行到，那也就是说 user 为 None，所以问题就变成了为什么 Django 的 authenticate 方法会返回 None，问题暂时转移到了 Django 层，我们继续追踪 Django 这部分的处理

def authenticate(self, request, username=None, password=None, **kwargs):
    if username is None:
        username = kwargs.get(UserModel.USERNAME_FIELD)
    try:
        user = UserModel._default_manager.get_by_natural_key(username)
    except UserModel.DoesNotExist:
        # Run the default password hasher once to reduce the timing
        # difference between an existing and a nonexistent user (#20760).
        UserModel().set_password(password)
    else:
        if user.check_password(password) and self.user_can_authenticate(user):
            return user

探索三

通过对 user 状态的追踪，我发现一直执行到 else 分支为止都是正常的，这时候发现一个可能的原因 usercanauthenticate，这个方法从名字来看是和权限相关的，很可能是这里拦截了未激活用户，查看了它的定义后发现确实如此

def user_can_authenticate(self, user):
    """
    Reject users with is_active=False. Custom user models that don't have
    that attribute are allowed.
    """
    is_active = getattr(user, 'is_active', None)
    return is_active or is_active is None

到这里就定位到问题了，但是因为 authenticate 是在库中调用的，所以似乎没有优雅的方法解决这个问题，我开始思索这个问题出现的原因，这应该是两个开源项目在升级过程中出现的兼容问题，所以这个锅该谁背？我查了一下这个方法的 git blame，发现它在 Django1.10 之前是不过滤未激活用户的，也就是说 Django 的版本更新中更改了这个方法的语义，这个时候我能想到的办法只有在项目中定义自己的验证方法，替换掉库中的这一部分。

def authenticate(self, request, username=None, password=None, **kwargs):
    if username is None:
        username = kwargs.get(UserModel.USERNAME_FIELD)
    try:
        user = UserModel._default_manager.get_by_natural_key(username)
    except UserModel.DoesNotExist:
        # Run the default password hasher once to reduce the timing
        # difference between an existing and a nonexistent user (#20760).
        UserModel().set_password(password)
    else:
        if user.check_password(password)
            return user

在做完这件事之后，我发现重新定义的方法和 Django 提供的只有关于未激活用户这一部分不同，Django 的开发者是否考虑到这个问题了呢？他们会怎么做？

结束

最后，我发现 jwt 库中会读取配置文件中的 authenticate backend 设定，因此再提供一个不过滤未激活用户的 backend 就可以解决这个问题了，Django 开发者是不是已经这么做了呢？在我换了关键词搜索后果然找到了，在 Django 文档中提及 AllowAllUserModelBackend，并且说明这个 backend 和正常的 backend 只有验证方法中是否过滤未激活用户这一点不同。这个问题的关键部分到这里就解决了。
