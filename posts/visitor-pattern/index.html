From FP to OOP, a.k.a Patterns

关于函数式与面向对象的争议自“古”有之，两种范式有各自擅长的领域，本文的目的是使通过函数式语言学会编程的人可以很好的将自己的编程思维transfer到面向对象语言上。

大多数使用函数式语言的人编程时主要都是在进行数据抽象和行为抽象，而最基本的抽象工具便是类型和模式匹配，下面举一个简单的Typed Racket的例子来阐释这点，

(define-type Expr (U Lit Add))
(struct Lit ([v : Integer]))
(struct Add ([v1 : Expr] [v2 : Expr]))

(: eval : Expr -> Integer)
(define (eval e)
  (match e
    [(Lit n) n]
    [(Add n1 n2) (+ (eval n1) (eval n2))]))

那么在面向对象语言中，这样的抽象应该如何表达呢？我们以Java为例:

interface Expr {
    int eval();
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }
    @Override
    public int eval() {
        return i;
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public int eval() {
        return e1.eval() + e2.eval();
    }
}

在Racket中我们分别定义了Lit，Add两种类型，然后称它们的并为Expr类型，也就是说我们有一个Union Type，而Lit，Add分别是它的subtype，函数eval通过模式匹配逐个handle Expr的subtype，并做出相应的处理。

那么Java的版本是如何做的呢？ 咱们定义了一个interface Expr， 以及两个实现了这个interface的类，Lit，Add，这相当于我们在Racket中定义的类型的并关系（Expr），那么基于Expr的行为抽象在Java中是如何实现的？我们在interface中声明了eval方法，并且在相应的类实现中重载了它，这种机制叫做Dynamic Dispatch，而这段Java代码展示是一种叫做interpreter pattern的设计模式。

这样似乎不错了，但是问题被解决了吗？ 如果我们再多写几个和eval相似的方法，比如查看类型的字符串表示，或者根据类型的信息对程序做一些更深入的变换，我们就得在每一个类中都加入对应的方法，这就带来了一个新的问题，它在软件工程中叫做耦合度太高，我们需要做的就是将数据与行为分离，靠什么呢？当然是我们在Java中的老朋友class，我们将上面的例子用数据与行为解耦的方式重写一次，

interface Expr {
    EvalV evalFn = new EvalV();
    int eval();
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }
    @Override
    public int eval() {
        return evalFn.forLit(i);
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public int eval() {
        return evalFn.forAdd(e1, e2);
    }
}

class EvalV {
    int forLit(int i) {
        return i;
    }
    int forAdd(Expr e1, Expr e2) {
        return e1.eval() + e2.eval();
    }
}

这段代码中，我们将所有eval相关的操作统一到EvalV这个类中，大写字母V表示这个类是一个Visitor类，如何使用这个类的呢？首先，我们在Expr接口中实例化EvalV类，得到evalFn这个对象，Fn是function的缩写，代表这是eval函数的面向对象版本，因为在接口中引入了evalFn这个对象，所以在每个实现了这个接口的类中，我们可以通过调用evalFn对应的“for + 类名”来实现eval的功能。

有没有发现EvalV和Racket中eval函数很相似，我们可以将for…这两个方法看做一个match，分别匹配Lit，Add两种类型，所做的操作也与Racket中的eval如出一辙，现在似乎有些点题了，我们可以直接将在函数式语言中的模式匹配写成Java中的一个Visitor类，而EvalV则可以看作一个Expr -> Int类型的函数。

// EvalV : Expr -> Int
class EvalV {
    int forLit(int i) {
        return i;
    }
    int forAdd(Expr e1, Expr e2) {
        return e1.eval() + e2.eval();
    }
}

如果前面的内容消化的差不多了，那么我们可以接下来的内容了，我们之前是如何使用EvalV这个类的？对，我们在接口中实例化EvalV为evalFn，但是这样又有一个问题了。如果我们有许多EvalV这样的Visitor类，在接口中会有许多evalFn这样的实例化操作，这个问题是不是似曾相识，没错，就是耦合的问题，为了解耦，我们需要将evalFn放到参数中，

interface Expr {
    int eval(EvalV evalFn);
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }

    @Override
    public int eval(EvalV evalFn) {
        return evalFn.forLit(i);
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public int eval(EvalV evalFn) {
        return evalFn.forAdd(e1, e2);
    }
}

class EvalV {
    int forLit(int i) {
        return i;
    }
    int forAdd(Expr e1, Expr e2) {
        return e1.eval(this) + e2.eval(this);
    }
}

比较这个版本和之前的版本有什么不同之处。首先是接口中的EvalV实例放到了eval方法的参数中，其次就是在EvalV中我们给eval方法传了个this参数，为了理解this在这里的作用，我们回顾一下eval的类型签名，EvalV -> int, 所以我们在调用的时候得给它传递一个EvalV的实例作为参数，那么this是什么，this指代了EvalV的对象本身，现在，请你给Expr添加一个View访问者类，用来查看类的字符串表示，类似于Java的toString方法，

interface Expr {
    int eval(EvalV evalFn);
    String view(ViewV viewFn);
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }
    @Override
    public int eval(EvalV evalFn) {
        return evalFn.forLit(i);
    }

    @Override
    public String view(ViewV viewFn) {
        return viewFn.forLit(i);
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public int eval(EvalV evalFn) {
        return evalFn.forAdd(e1, e2);
    }

    @Override
    public String view(ViewV viewFn) {
        return viewFn.forAdd(e1, e2);
    }
}

class EvalV {
    int forLit(int i) {
        return i;
    }
    int forAdd(Expr e1, Expr e2) {
        return e1.eval(this) + e2.eval(this);
    }
}

class ViewV {
    String forLit(int i) {
        return "" + i + "";
    }
    String forAdd(Expr e1, Expr e2) {
        return e1.view(this) + " + " + e2.view(this);
    }
}

添加ViewV访问者类的过程大抵和EvalV相同，不过你应该会注意到，在Lit，Add中，eval和view方法和相似，因为形参名可以为任意，所以我们可以统一形参名，

    @Override
    public int eval(EvalV Fn) {
        return Fn.forLit(i);
    }

    @Override
    public String view(ViewV Fn) {
        return Fn.forLit(i);
    }

但是这样并没有解决问题，因为eval和view的类型不相同，eval是EvalV -> Int，而view是ViewV -> String，如果想要让这两个方法具有相同的类型，应该怎么做？我们可以使用Object来代替Int和String，这种做法叫做放宽约束，因为任何类都是Object的子类，同理在静态类型函数式语言中，我们可以使用Any类型，因为任何类型都是Any的subtype

    @Override
    public Object eval(EvalV Fn) {
        return Fn.forLit(i);
    }

    @Override
    public Object view(ViewV Fn) {
        return Fn.forLit(i);
    }

但是参数类型还是不相同，我们也得给它放宽约束，怎么做呢？我们知道Java中可以通过继承来实现，因为任何类都隐式的继承Object类，所以我们可以定义一个abstract class，然后让EvalV和ViewV都继承于它，但是也可以使用我们之前定义Expr类型的方法，定义一个ExprVisitorI接口，然后让EvalV和ViewV都实现它，这对应于Typed Racket中的Union Type，同样为了点题，我们使用第二种方法，

interface Expr {
    Object eval(ExprVisitorI Fn);
    Object view(ExprVisitorI Fn);
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }
    @Override
    public Object eval(ExprVisitorI Fn) {
        return Fn.forLit(i);
    }

    @Override
    public Object view(ExprVisitorI Fn) {
        return Fn.forLit(i);
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public Object eval(ExprVisitorI Fn) {
        return Fn.forAdd(e1, e2);
    }

    @Override
    public Object view(ExprVisitorI Fn) {
        return Fn.forAdd(e1, e2);
    }
}

interface ExprVisitorI {
    Object forLit(int i);
    Object forAdd(Expr e1, Expr e2);
}

class EvalV implements ExprVisitorI {
    public Object forLit(int i) {
        return i;
    }
    public Object forAdd(Expr e1, Expr e2) {
        return (Integer) e1.eval(this) + (Integer) e2.eval(this);
    }
}

class ViewV implements ExprVisitorI {
    public Object forLit(int i) {
        return "" + i + "";
    }
    public Object forAdd(Expr e1, Expr e2) {
        return e1.view(this) + " + " + e2.view(this);
    }
}

现在我们可以将eval和view方法统一了，为什么？因为他们的方法体和类型都相同，换句话说只有名字不同， 那么统一后，我们应该给它一个更通用的名字，按照惯例，将它命名为accept，最终代码如下：

interface Expr {
    Object accept(ExprVisitorI Fn);
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }
    @Override
    public Object accept(ExprVisitorI Fn) {
        return Fn.forLit(i);
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public Object accept(ExprVisitorI Fn) {
        return Fn.forAdd(e1, e2);
    }
}

interface ExprVisitorI {
    Object forLit(int i);
    Object forAdd(Expr e1, Expr e2);
}

class EvalV implements ExprVisitorI {
    public Object forLit(int i) {
        return i;
    }
    public Object forAdd(Expr e1, Expr e2) {
        return (Integer) e1.accept(this) + (Integer) e2.accept(this);
    }
}

class ViewV implements ExprVisitorI {
    public Object forLit(int i) {
        return "" + i + "";
    }
    public Object forAdd(Expr e1, Expr e2) {
        return e1.accept(this) + " + " + e2.accept(this);
    }
}

最后，让我们试试解决更有趣的问题，由于我们已经将数据和行为分离了，所以在Java解决这个问题和在Typed Racket中是很相似的，我们也需要拓展访问者类去handle新的子类，这在Java中是很容易实现的，因为interface是可以被继承的，所以我们可以定义一个MulExprVisitorI，这个新的interface不仅继承原有的forLit，forAdd方法，还拓展了forMul方法，所以我们可以定义MulEvalV和MulViewV分别继承EvalV和ViewV，并添加forMul方法。

interface Expr {
    Object accept(ExprVisitorI Fn);
}

class Lit implements Expr {
    int i;
    Lit(int _i) {
        i = _i;
    }
    @Override
    public Object accept(ExprVisitorI Fn) {
        return Fn.forLit(i);
    }
}

class Add implements Expr {
    Expr e1, e2;
    Add(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public Object accept(ExprVisitorI Fn) {
        return Fn.forAdd(e1, e2);
    }
}

class Mul implements Expr {
    Expr e1, e2;
    Mul(Expr _e1, Expr _e2) {
        e1 = _e1;
        e2 = _e2;
    }

    @Override
    public Object accept(ExprVisitorI Fn) {
        return ((MulExprVisitorI)Fn).forMul(e1, e2);
    }
}

interface ExprVisitorI {
    Object forLit(int i);
    Object forAdd(Expr e1, Expr e2);
}

interface MulExprVisitorI extends ExprVisitorI {
    Object forMul(Expr e1, Expr e2);
}

class EvalV implements ExprVisitorI {
    public Object forLit(int i) {
        return i;
    }
    public Object forAdd(Expr e1, Expr e2) {
        return (Integer) e1.accept(this) + (Integer) e2.accept(this);
    }
}

class MulEvalV extends EvalV implements MulExprVisitorI {
    public Object forMul(Expr e1, Expr e2) {
        return (Integer) e1.accept(this) * (Integer) e2.accept(this);
    }
}

class ViewV implements ExprVisitorI {
    public Object forLit(int i) {
        return "" + i + "";
    }
    public Object forAdd(Expr e1, Expr e2) {
        return e1.accept(this) + " + " + e2.accept(this);
    }
}

class MulViewV extends ViewV implements MulExprVisitorI {
    public Object forMul(Expr e1, Expr e2) {
        return e1.accept(this) + " * " + e2.accept(this);
    }
}

到这里本文就基本结束了，接下来如果你对Java感兴趣可以读一读Java编程思想一类的书，用Java写一些实际点的项目，比如一个DB，或许你就会发现为什么Java在工业界占据如此重要的地位，当然你也可以在Python之类的语言中使用访问者模式，事实上很多著名的开源项目都使用了这个模式，比如Rails项目中的arel，这是Rails中Active Record用来构造SQL查询的DSL*，当然本文并没有提及全部FP的惯用法所对应的模式，除了比较简单之外，也是因为访问者模式是这种transfer的核心，其余的模式大多围绕它展开（Factory Pattern etc），感兴趣的读者可以继续阅读A Little Java, A Few Patterns或者CS2510的课件。
